# next-codex-workflow v0.1.2 테스트 리포트

## 개요

이 문서는 `next-codex-workflow` `v0.1.2` 기준으로 수행한 실제 세션 로그 비교 결과를 정리한 내부 테스트 리포트다.

이번 비교의 목적은 다음 두 가지였다.

- `single-agent` 워크플로우와 `multi-agent` 워크플로우의 실제 토큰 사용량 차이 확인
- 멀티 에이전트 구조가 오케스트레이터의 메인 컨텍스트를 더 오래 깨끗하게 유지하는지 확인

이 결과는 패키지가 아직 개발 중이지만, 이미 실제 프로젝트 수준에서 의미 있는 결과를 만들고 있다는 근거로 포트폴리오에 활용할 수 있다.

## 비교 대상

비교에 사용한 세션 로그:

- 멀티 에이전트 세션
  - `session_logs/25/rollout-2026-03-25T14-05-02-019d2361-c405-7693-ada9-417ec23567d9.jsonl`
- 싱글 에이전트 세션
  - `session_logs/25/rollout-2026-03-25T14-17-14-019d236c-edd6-7e43-a924-0483be7d4a91.jsonl`

두 세션 모두 아래 4개 기능 청크를 처리했다.

- `PK-001` Home Experience
- `PK-012` Product Visual System
- `PK-013` Data Normalization Layer
- `PK-002` Pokedex Browse

즉, 비교 범위는 완전히 동일한 축으로 볼 수 있다.

## 핵심 결과

가장 중요한 결과는 다음과 같다.

- 총 토큰 사용량은 거의 비슷했다.
- 하지만 멀티 에이전트는 메인 오케스트레이터 스레드의 출력, 추론, 툴 실행량을 크게 줄였다.
- 즉, 멀티 에이전트의 핵심 장점은 "무조건 더 빠르다"가 아니라 "메인 컨텍스트를 더 오래 깨끗하게 유지한다"는 점으로 보인다.

## 멀티 에이전트 세부 분해

멀티 에이전트 세션에서는 총 `38`회의 서브에이전트 실행이 발생했다.

- `explorer`: 5회
- `planner`: 4회
- `executor`: 6회
- `tester`: 6회
- `verifier`: 10회
- `reviewer`: 7회

이 분해는 `v0.1.2`에서 도입된 워크플로우 구조가 실제로 동작했음을 보여준다.

## 해석

이번 비교에서 멀티 에이전트가 보여준 가장 강한 결과는 총 토큰 절감 그 자체보다도, 메인 스레드의 잡음 감소다.

- 메인 스레드 출력 토큰 `56.1%` 감소
- 메인 스레드 추론 토큰 `61.9%` 감소
- 메인 스레드 tool 호출 `39.8%` 감소

이 숫자는 다음 가설을 뒷받침한다.

- 멀티 에이전트 구조는 전체 작업을 더 작은 역할 단위로 분산시켜 오케스트레이터가 요구사항, 의사결정, 승인 흐름을 더 오래 유지하게 해준다.
- 싱글 에이전트 구조는 같은 범위를 처리하더라도 메인 스레드 하나에 탐색, 계획, 구현, 검증, 리뷰가 모두 쌓이기 때문에 컨텍스트 오염이 더 빨리 발생할 가능성이 높다.

즉, 현 시점에서 이 패키지의 주된 강점은 "속도"보다 "컨텍스트 위생"과 "오케스트레이션"에 있다.

## 시간 데이터 해석 시 주의점

세션 로그 기준 경과 시간은 다음과 같았다.

- 멀티 에이전트: 약 `8시간 11분`
- 싱글 에이전트: 약 `1시간 46분`

하지만 이 수치는 공정한 성능 벤치마크로 사용하기 어렵다.

이유:

- 멀티 에이전트 세션에는 더 많은 사용자 턴이 포함되어 있다.
- 멀티 에이전트 세션에는 `turn_aborted`가 3회 존재한다.
- 사용자가 중간에 재개/재시도/커밋 요청을 여러 번 넣었기 때문에 순수 실행 시간 비교로 보기 어렵다.

따라서 이번 리포트에서는 시간보다 토큰 구조와 메인 스레드 부담 감소를 핵심 지표로 보는 것이 더 정직하다.

## 포트폴리오용 문장 예시

### 짧은 버전

`next-codex-workflow`라는 에이전트 오케스트레이션 기반 Next.js 개발 워크플로우 패키지를 설계하고 실험했다. 실제 PokeAPI 프로젝트에서 동일한 기능 청크를 기준으로 단일 스레드와 멀티 에이전트 워크플로우를 비교한 결과, 총 토큰 사용량은 거의 비슷하게 유지하면서도 멀티 에이전트 방식이 메인 스레드 출력 토큰을 56%, 추론 토큰을 62%, 툴 호출을 40% 가까이 줄였다.

### 긴 버전

멀티 에이전트 기반 개발 워크플로우 패키지 `next-codex-workflow`를 직접 설계·구현하고, PokeAPI 기반 Next.js 프로젝트에서 `Home`, `Visual System`, `Data Normalization`, `Pokedex Browse`의 4개 기능 청크를 대상으로 `single-agent`와 `multi-agent` 방식을 비교 검증했다. 그 결과 총 토큰 사용량은 거의 동일한 수준을 유지하면서도, 멀티 에이전트 구조는 오케스트레이터 메인 스레드의 출력 토큰을 56.1%, 추론 토큰을 61.9%, shell/tool 호출을 39.8% 줄여 장기 작업에서 더 깨끗한 메인 컨텍스트를 유지하는 경향을 보였다.

## 현재 시점 결론

`v0.1.2`는 아직 개발 중인 도구지만, 이미 다음과 같은 결과를 보여주고 있다.

- 실제 기능 단위 개발에 사용 가능
- 계획, 구현, 테스트, 검증, 리뷰 역할 분리가 실제 세션에서 동작함
- 멀티 에이전트 사용 시 총 토큰이 폭증하지 않음
- 멀티 에이전트의 강점이 메인 스레드 컨텍스트 보호라는 형태로 관측됨

따라서 이 프로젝트는 아직 완성형은 아니지만, "에이전트 오케스트레이션 기반 개발 워크플로우"라는 방향성이 실제 데이터로 검증되기 시작한 단계라고 볼 수 있다.
